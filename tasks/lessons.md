# Lessons

- Failure mode: treating `dvm start` / live-VM validation as an ordinary verification step while the user already has an active VM they care about.
- Detection signal: the user explicitly warned `DO NOT stop it under any circumstances when you start testing this`.
- Prevention rule: before any DVM runtime verification, check whether a live VM is in use; default to `nix eval`, direct script/config inspection, and other non-disruptive proofs unless the user explicitly authorizes touching the running VM.

- Failure mode: assuming method-call-style attribute access precedence in Nix (`foo.bar { ... }.config`) when extending evaluated systems for synthetic proofs.
- Detection signal: the synthetic `extendModules` proof failed with `attribute 'config' missing` until the function call was parenthesized.
- Prevention rule: when calling a function stored in an attribute and then accessing fields on the result, parenthesize the call explicitly: `(foo.bar { ... }).config`.

- 2026-04-02: When a VM-side network stack depends on a static DHCP lease, configure it from the actual VM NIC MAC produced by `VMConfigurator`, not a guessed/default value. Detection signal: the guest gateway/DHCP identity looks wrong or the guest gets the wrong subnet/IP. Prevention rule: create the VM first, then pass the real NIC identity into the sidecar/network config.
- 2026-04-02: Do not verify mounts by grepping shell-quoted paths or assuming guest-visible `/var/...` paths appear unchanged in `mount` output on macOS. Detection signal: mounts succeed functionally but verification logs `unexpected fs type` or `MISSING`, followed by `Resource busy` on retry. Prevention rule: verify using raw paths, include `/private` aliases for macOS paths, and add a short post-mount settle check before declaring failure.
- 2026-04-07: Lint-driven refactors on the guest RPC path must preserve blocking `net.Listener` semantics and be validated with a full `dvm start` boot. Detection signal: activation succeeds, but the host agent proxy logs repeated HTTP/2 preface resets while the guest RPC daemon flaps with `accept ... resource temporarily unavailable`. Prevention rule: treat transport/socket code as behavior-sensitive, keep guest daemon logs visible through the shared state directory, and run an end-to-end start/stop cycle after any agent/vsock refactor.
- 2026-04-10: When a user prioritizes ergonomics over architectural purity, do not split a configuration model across multiple files without clear evidence it is worth the extra indirection. Detection signal: the proposed design needs an additional config file or cross-file lookup to express a single user concept. Prevention rule: first try a single-file schema with explicit typed fields and sensible defaults, then add indirection only if the simpler model demonstrably fails.
- 2026-04-10: Any harness or helper that stages files into `~/.local/state/dvm` must recreate its subdirectories after `dvm start`, because startup wipes and recreates the state dir. Detection signal: a post-boot copy into `~/.local/state/dvm/...` fails with `No such file or directory` even though the parent was created earlier in the script. Prevention rule: treat the state dir as boot-scoped runtime state, not persistent setup, and create staging directories immediately before writing into them.
- 2026-04-10: Do not rely on backslash-newline shell continuations inside nested `sh -lc '...'` strings when the command passes through multiple quoting layers. Detection signal: the guest shell reports `/bin/sh:  : command not found` even though each individual clause looks valid. Prevention rule: flatten guest command strings to single lines or generate them explicitly instead of using line continuations inside quoted shell fragments.
- 2026-04-10: When a top-level dispatcher accepts control flags plus passthrough flags, strip the control flags before invoking the child command. Detection signal: a child script rejects a mode flag like `--e2e` or `--all` that should have been consumed by the parent. Prevention rule: parse flags into distinct buckets for dispatch mode versus child passthrough, and verify the rendered command with a dry run.
- 2026-04-10: Do not treat nested inline `sh -lc '...$VAR...'` assertions as proof of guest runtime behavior when the command crosses multiple host and guest shell layers. Detection signal: a harness assertion reports an exported variable as empty even though the sourced env file and downstream behavior both show it is present. Prevention rule: use mounted probe scripts or other file-based probes for shell/runtime assertions instead of fragile nested one-liners.
- 2026-04-10: HomeLink install now treats pre-existing non-empty guest home-mount directories as recoverable by moving them to a timestamped backup and proceeding with a symlink, to avoid hard-stop boot failures when mount state persists across sessions (e.g. .pi/agent).
- 2026-04-10: `dvm shell` and `dvm exec` should treat missing/empty credential environment values as non-fatal to avoid blocking user workflows. Detection signal: commands fail early with `environment variable '<VAR>' is not set` before launching the target process. Prevention rule: keep missing host env credentials warning-only for recoverable cases, but still inject any resolvable proxy/passthrough credentials.
- 2026-04-10: Do not let a reusable flake module export or depend on a project-specific registry convention when standard flake outputs already express the concept. Detection signal: a supposedly general-purpose flake-parts module writes to a custom namespace like `flake.modules.*` instead of standard outputs such as `flake.darwinModules`. Prevention rule: make generic flakes expose standard `darwinModules`/`flakeModules.default` surfaces first, and keep project-specific wiring only in downstream consumers.
- 2026-04-11: Do not run ad-hoc local smoke tests that depend on background listeners without an explicit timeout and known-good teardown path. Detection signal: a verification command produces no output for an unusual amount of time or the session has to be interrupted to clean up a stuck background process. Prevention rule: wrap listener/load-generator smoke tests in `timeout`/bounded waits, capture child PIDs explicitly, and prove the teardown command works before starting the test.
- 2026-04-11: Long-running harness waits and cleanup paths must emit heartbeats so a human or agent can tell the run is still progressing. Detection signal: the harness sits silent for tens of seconds during phases like waiting for port forwarding, stress execution, or artifact collection, leaving the operator unsure whether it hung. Prevention rule: add periodic progress logs to every wait loop and log cleanup sub-steps explicitly, especially when waiting for shutdown or crash-report collection.
- 2026-04-11: Verification in a harness must use the same privilege context as the production code path it is trying to prove. Detection signal: a harness says a service is missing while the supervised process is clearly running, especially when the service runs as root or under launchd. Prevention rule: match the real code path's command and privileges exactly (for example `sudo lsof` if the agent runs the listener scan as root) instead of probing from an unprivileged shell.
- 2026-04-11: Do not hide a background process launch behind command substitution in shell helpers. Detection signal: a step like "boot vm" prints once and then appears frozen before any heartbeat logs, because the caller waits for the helper's subshell to finish instead of getting the spawned PID immediately. Prevention rule: when a helper launches long-lived background work, assign the PID via a variable name / nameref in the current shell (or launch directly in the caller), and rerun the affected harness end-to-end after extracting shared lifecycle helpers.
- 2026-04-11: VM-backed verification on macOS must use an entitled `dvm-core` build path, not a SwiftPM test binary. Detection signal: `dvm start` fails immediately with `Invalid virtual machine configuration` and a missing `com.apple.security.virtualization` entitlement after a host-side `swift test` or fast test run. Prevention rule: before any VM-backed test/probe, rebuild via the project’s entitled path (`just build` or the harness helper that wraps it) and do not assume a freshly compiled SwiftPM binary is runnable under Virtualization.
- 2026-04-12: Do not label network relay close-noise suppression as "safe" without checking whether the same error strings can also be the first observable symptom of a real bug. Detection signal: a user asked whether suppressing `broken pipe` / `reset by peer` / closed-network errors was 100% safe, and the existing `isConnCloseError` comment already documented that premature proxy teardown bugs can produce the same messages. Prevention rule: distinguish first-failure signals from secondary teardown noise before suppressing logs; only suppress follow-on close errors or keep a diagnostic escape hatch.
- 2026-04-12: When adding scoped lint support for a CLI tool, verify the tool's argument model instead of assuming it accepts multiple file paths like other linters. Detection signal: `just lint` failed with `statix check` rejecting multiple changed-file arguments. Prevention rule: smoke-test the scoped lint path against the actual changed file set and, for less-familiar tools, run them in a per-file loop unless the CLI explicitly supports batching.
- 2026-04-12: Terminal color passes should stay conservative; not every informational line benefits from emphasis. Detection signal: the user called out that lines like `The boot script may be incompatible with this version of dvm-core.` and `MITM CA installed in guest trust store.` should remain plain because the added color made them noisier, not clearer. Prevention rule: default to coloring only true status markers, identifiers, and warnings/errors; if a line reads as explanatory or routine confirmation, leave it uncolored unless there is clear evidence that emphasis improves scanability.
- 2026-04-12: Before adding new guest/runtime marker env vars, check whether DVM already exports a stable environment identity marker that downstream code can consume directly. Detection signal: the Pi runtime-notice design initially added `PI_RUNTIME_*` plumbing even though guest plumbing already sets `DVM_GUEST=1`. Prevention rule: prefer existing guest env contracts like `DVM_GUEST` over inventing redundant wrapper-specific markers.
